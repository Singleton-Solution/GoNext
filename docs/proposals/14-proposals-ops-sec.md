## Doc 11 — Testing & CI

### Q11-1: Plugin marketplace CI compute budget
**Source**: doc 11 §20 (running the contract suite on every third-party upload is unbounded compute cost).

**Proposal**: Adopt three concrete caps with no money attached in v1.
1. **Per-check wall-clock time-box: 10 minutes.** Anything that runs longer fails the submission with `timeout`. Plugin contract suite hits ~2 min today (§7); 10 min absorbs a 5× regression.
2. **Per-author submission rate-limit: 5 successful submissions / day; 10 failed / day with exponential cooldown** (1 min, 2 min, 4 min, ... capped at 1 hour) after each fail. Author identity = the OIDC-bound publisher identity from §13.4.
3. **Per-bundle dedup**: re-submitting the same bundle hash within 24 h returns the cached result without re-running the suite.

No per-check billing in v1. Add billing only if abuse persists after the rate-limit lands.

**Reasoning**: Time-box + rate-limit handles 99% of accidental waste (CI loops on broken bundles) without revenue infrastructure. Billing is a much heavier lift than caps; defer it. The cooldown shape penalizes a stuck author without locking them out.

**Confidence**: high
**Reversibility**: cheap

### Q11-2: WP REST corpus refresh cadence
**Source**: doc 11 §20 (when and from which WP versions do we re-record).

**Proposal**: Pin the corpus to the **latest two WP minor releases at any time**. Refresh **on each new WP minor release**, automated via a nightly job that detects a new WP version on wordpress.org/releases.txt and opens a PR with the re-recorded corpus and a diff report. Old minor drops out of corpus when the new one lands. Hand-review the PR diff before merge.

**Reasoning**: Two minors covers the bulk of the WP install base at any one time (WP users upgrade fast enough that anything older is sub-1%). Automation removes the "we forgot to refresh" failure mode. PR-gated keeps a human in the loop for unexpected schema changes that would otherwise silently green-light migration breakage.

**Confidence**: high
**Reversibility**: cheap

---

### Q11-3: Visual regression baseline ownership
**Source**: doc 11 §20 (engineering vs design review of visual diffs).

**Proposal**: **Engineering owns the baselines**. Design is added as a non-blocking reviewer on PRs that touch the visual-regression directory for shared components (header, post list, editor chrome). Baselines update via `make test-update`, same as golden files. Keep visual regression opt-in via `@visual` tag until **P5** (post-GA polish phase).

**Reasoning**: Visual-regression tooling is engineering ergonomics; design's value-add is on the un-captured "does it look right" dimension which we don't gate CI on. Adding design as a blocking reviewer would have made PR latency the bottleneck. Reserving the wider rollout for P5 means we are not paying baseline-churn cost during the high-velocity build phase.

**Confidence**: medium
**Reversibility**: cheap

### Q11-4: Theme contract suite versioning
**Source**: doc 11 §20 (should the suite be versioned, themes opt in by `apiVersion`).

**Proposal**: **Yes, version the suite.** Themes declare `apiVersion: "1"` (etc.) in `theme.json`. We support the last **two major versions** of the suite at any time. A new theme suite major drops:
- N most-recent: supported until N+2 ships.
- The marketplace publish API rejects submissions on a deprecated major one month before sunset, with a migration guide diff.

Inside a major, new checks are additive only — they go in as `warning` for one minor and become `error` in the next.

**Reasoning**: Themes are a third-party ecosystem; each new contract rule is a breaking change without versioning. The "two-major support" rule mirrors how Chrome / WordPress / Node do API deprecations and gives authors a predictable window. The additive-then-promoting cadence makes "your theme started failing CI this morning" a thing that only happens after a warning period.

**Confidence**: high
**Reversibility**: moderate (once `apiVersion` is in the manifest schema, removing it requires a manifest migration)

### Q11-5: Migration corpus IP / licensing
**Source**: doc 11 §20 (10-site corpus contains real anonymized WP content; licensing).

**Proposal**: **Replace the real-anonymized corpus with a fully synthetic corpus generated by a "WP-shape generator" before P3 launch.** The generator produces sites whose distribution of post counts, taxonomy depth, plugin density, shortcode variety, embedded oEmbed types, and HTML pathology matches what we observed in the real sites, **but no content is sourced from any real install**. Keep the 10-site shape catalog as a YAML spec; the corpus is regenerated from it deterministically.

Real sites can still be used in **ad-hoc maintainer-only** testing under a signed-evaluator agreement, never committed.

**Reasoning**: Anonymization is necessary but not sufficient — re-identification of authors via writing style and metadata is well-documented. Synthetic data eliminates the licensing question entirely. The shape catalog preserves the testing signal because what matters for an importer is HTML-pathology variety, not literal content. This also makes the corpus extensible (add more synthetic shapes as we find new pathology).

**Confidence**: high
**Reversibility**: moderate (the shape catalog + generator is real engineering work)

### Q11-6: Mutation testing promoted to required
**Source**: doc 11 §20 (if mutation testing proves valuable on critical packages, do we make it a gate).

**Proposal**: **Do not promote.** Keep `go-mutesting` and Stryker informational, exactly as today. The mutation-score number is a tooltip on a maintainer dashboard, not a CI gate.

**Reasoning**: The doc itself names the failure mode (chasing the score writes brittle tests). The marginal value of mutation testing for the `policy`/`plugin`/`migrations` packages is signal that tests-could-be-better; that signal is already obtained by reading the score. Turning it into a gate trades the signal for compliance theater. If a package's mutation score drops dramatically, the package owner can investigate — they don't need a red CI to be told.

**Confidence**: high
**Reversibility**: cheap

### Q11-7: Property-based testing for plugin host
**Source**: doc 11 §20 (random plugin manifests + WASM-ish modules to fuzz the host).

**Proposal**: **Spike in P4 (2-week timebox).** If the spike finds a host-instantiator bug the enumerated capability matrix missed, ship it as a `gonext plugin fuzz` CLI users can opt into and add a nightly informational fuzz job in CI. If the spike finds nothing not already covered, archive and don't pursue.

**Reasoning**: Property-based testing has a clear cost-benefit curve in this codebase already — strong for parsers and pure transforms (§19.5), weak for CRUD code. The plugin host sits in the middle; generating valid-ish manifests is non-trivial. A timeboxed spike is the right call to find out which side this lands on without committing engineering time upfront.

**Confidence**: medium
**Reversibility**: cheap

### Q11-8: Hot-path budget edit policy
**Source**: doc 11 §20 (who can edit the hot-path list).

**Proposal**: Edits to `perf/budgets.yaml` require:
1. A PR with no other source changes (budget edits are their own PR class).
2. CODEOWNERS approval from `@perf-leads` (a small named group: 2 engineers + 1 SRE).
3. A line in the release notes for the relevant cycle.

No retroactive loosening to land a feature PR. If a feature regresses a budget, the feature gets reworked or the budget edit goes through this gate as its own decision.

**Reasoning**: The risk is exactly the "PR loosens budget to land feature" pattern. Separating the budget edit into its own PR makes it visible; CODEOWNERS makes it owned; release-notes makes it accountable. Three lightweight rules turn an editorial document into a governance artifact without the overhead of a full change-control process.

**Confidence**: high
**Reversibility**: cheap

### Q11-9: Local stack "lite" profile (sqlite + in-process Asynq)
**Source**: doc 11 §20 (docker-compose is ~3 GB; lite profile worth it?).

**Proposal**: **Do not build a lite profile.** Instead, invest the same engineering time in:
1. Faster docker-compose boot via pinned image digests + buildkit cache + `tmpfs` data volumes for ephemeral state.
2. A `--minimal` flag on `gonext dev` that disables the `worker-migrate` replica and the second Next.js process (admin shares one Next.js with public) — drops footprint to ~1.8 GB without forking the test surface.

**Reasoning**: A lite profile (sqlite + in-process Asynq) is a second product. Every test that touches a Postgres-specific feature (§19.1's whole point) has to be skipped or stubbed in lite; every divergence becomes a "passed in lite, failed in prod" bug. Doc 11 already rejected sqlmock for the same reason — that argument extends to a sqlite dev mode. The 8 GB-laptop developer is real but solvable with a footprint-reduction pass on the real stack.

**Confidence**: medium
**Reversibility**: moderate (once a lite mode ships, removing it is socially expensive)

### Q11-10: Cross-cutting flake triage ownership
**Source**: doc 11 §20 (when refactors collide on a flaky e2e, who triages).

**Proposal**: Until team size >= 8 engineers: **the engineer on rotation as on-call triages flakes during their on-call week.** Filing the flake against the owning subsystem CODEOWNER is part of the on-call job. Above 8 engineers: weekly named-rotation independent of on-call, posted in the sprint kickoff.

The flake-detection mechanism is automatic: any test that passes on a retry is tagged `flake:<sha>` in the test report; the on-call's daily digest includes that day's tags.

**Reasoning**: The doc identifies the right answer ("weekly rotation, only viable once headcount supports it"). Folding flake triage into on-call until then avoids a second rotation that nobody owns. The "flake on retry" detection is already implied by §11.3's retry policy — make it observable in the digest.

**Confidence**: high
**Reversibility**: cheap

## Doc 12 — Jobs & Cron

### Q12-1: Cross-region cron scheduler
**Source**: doc 12 §20 (single Redis means single-region scheduler; multi-region survival).

**Proposal**: **Defer to P5** (when multi-region deployment work begins). The single-region scheduler is acceptable until then because:
- Cron tasks tolerate missed ticks (§8.4 covers this).
- The "primary Redis fails" path is already covered by Asynq's lease expiry → re-elect within 30 s.

Document the single-region failure mode in `runbooks/cron-leader-missing.md` so on-callers know to expect cron quiescence during a primary Redis outage. Resolution trigger: the first design pass on multi-region deployment.

**Reasoning**: Cross-region coordination is a fundamentally different problem (CAP, partition tolerance, lease semantics under network partitions). Designing it before there's a real second region is speculative; designing it once the second region's traffic shape is concrete is grounded. The current design's failure mode is well-bounded.

**Confidence**: high
**Reversibility**: moderate (the chosen coordination primitive shapes the migration)

---

### Q12-2: Per-tenant queues vs shared queues with per-tenant rate limits
**Source**: doc 12 §20 (when multisite lands, do we partition queues by tenant).

**Proposal**: **Shared queues with per-tenant rate limits + per-tenant metrics.** Same shape as per-plugin scoping (§4.2): a token-bucket per tenant per queue, defaults set by tenant plan tier, observable via `jobs_enqueued_total{tenant,queue}`.

Revisit only if a real workload shows starvation that rate-limits can't address (e.g., one tenant's tasks have a p99 latency that crowds out another tenant's p50). At that point, a hot-queue split (one extra `default-large-tenant` queue, not thousands of per-tenant queues) is the upgrade path.

**Reasoning**: Asynq's per-queue weight system is not built for thousands of queues; the doc itself names this (§19.6). Per-tenant rate limits accomplish the fairness goal (no tenant can monopolize) without paying the structural cost. The "hot tenant gets a dedicated queue" upgrade is a well-known pattern that we can apply targeted, not blanket.

**Confidence**: high
**Reversibility**: moderate (once tenants depend on per-tenant queue names in their integrations, undoing it is expensive)

---

### Q12-3: Priority preemption
**Source**: doc 12 §20 (cooperative preemption for `critical` saturation).

**Proposal**: **Do not add preemption.** If `critical` ever saturates, the answer is to size its concurrency larger (it defaults to 4 — the doc itself notes the priority matters more than throughput). At sustained saturation the alert from `jobs_queue_depth{queue="critical"} > 50 for 5min` is the page.

**Reasoning**: Cooperative preemption in Go is invasive (every handler grows a yield check), and impossible in WASM-dispatched plugin jobs without ABI changes. The benefit is contingent on a saturation that the current design treats as a sizing bug, not a runtime bug. The doc's own §19.7 rejected this for v1 — that rejection is correct for v2 too.

**Confidence**: high
**Reversibility**: cheap

---

### Q12-4: Worker autoscaling
**Source**: doc 12 §20 (autoscaler policy is undesigned).

**Proposal**: In P4, ship a Kubernetes HPA configured on `jobs_queue_depth{queue}` for **two queues only: `media` and `migrate`**.
- `media`: scale 1 → 8 replicas when `jobs_queue_depth{queue="media"} / replica_count > 50` for 2 min; scale down with 10 min cooldown.
- `migrate`: scale 0 → 2 when depth > 0 for 1 min; scale down to 0 when depth = 0 for 30 min.

Do not autoscale `critical`/`default`/`cache`/`low`/`plugins` — their workload is small per task and concurrency is set high enough per replica that a static replica count handles the peak.

**Reasoning**: Autoscaling pays off where work is bursty and CPU-bound (`media`) or campaign-shaped (`migrate`). For latency-sensitive small-task queues, autoscaling adds boot-time latency without solving anything. The narrow rollout means we ship autoscaling where it has clear ROI and skip it where it would be cargo cult.

**Confidence**: medium
**Reversibility**: cheap

### Q12-5: Transactional enqueue
**Source**: doc 12 §20 (Redis-vs-Postgres consistency for enqueues in DB transactions).

**Proposal**: **Adopt the transactional outbox pattern.** A `pending_tasks` table (Postgres) receives a row in the same transaction as the business mutation; a relay process drains the table into Asynq with at-least-once semantics. Default-on for `webhook.deliver`, `email.password_reset`, `email.verify`, `email.send`. Opt-in flag per enqueue call for everything else.

The relay is itself a worker process (`worker-outbox`) that polls `pending_tasks WHERE relayed_at IS NULL ORDER BY id LIMIT 1000`, enqueues, sets `relayed_at`. Idempotency via the existing `idem_key` system handles double-fire if the relay crashes mid-batch.

Replaces the `pending_critical_tasks` table sketched in §19.1 with a more general primitive.

**Reasoning**: The class of bug the question asks about (TX commits + Redis write fails) is real and only obvious in incident postmortems. The outbox pattern is well-established (Microservices.io and dozens of production reports); the per-task overhead is one Postgres INSERT, which is cheap relative to the work being enqueued. Default-on for the most consequential tasks (external-facing webhooks and emails) gets us the "we never lose a password-reset email" story without an opt-in burden everywhere.

**Confidence**: high
**Reversibility**: moderate (once the relay exists, removing it is a TX-semantics regression)

### Q12-6: Plugin WASM old-version cleanup
**Source**: doc 12 §20 (old plugin module memory after upgrade).

**Proposal**: **LRU evict idle plugin modules** from worker memory after **30 min of zero dispatches**. Implementation: the plugin manager tracks last-dispatch-time per (slug, version, replica); a sweeper goroutine evicts entries past the threshold. New dispatches re-instantiate from the bundle cache (which itself is a per-replica LRU bounded by total bytes).

Emit `plugin.module.evicted_total{slug,version,reason="idle"|"capacity"}`.

**Reasoning**: 30 min covers the realistic upgrade-and-quiesce window without making active plugins pay re-instantiation cost. A capacity-based LRU is the right primitive because a busy site with many plugins is the constraint, not time-since-upgrade. The eviction metric makes the policy observable so we can tune the threshold from data.

**Confidence**: medium
**Reversibility**: cheap

### Q12-7: DLQ payload privacy retention
**Source**: doc 12 §20 (raw payloads in Redis archive for 14 days; shorter retention or encryption).

**Proposal**: **Both, scoped by task-spec flag.**
1. Add `Sensitive: bool` to `TaskSpec`. Tasks with `Sensitive: true` (initially: `email.password_reset`, `email.verify`, `webhook.deliver` body, anything carrying a token_ref) have their payload **encrypted at rest** in the archive using a system-secret-derived key (separate from the pepper; rotated independently).
2. Sensitive tasks have a **3-day DLQ retention** instead of the default 14.

The admin UI's "Inspect payload" decrypts on read, audit-logs the access.

**Reasoning**: 14 days of plaintext password-reset tokens in Redis is a real exposure even with §6.1 redaction (the redaction only protects the UI; an attacker with Redis access reads raw). Encryption-at-rest moves the threat from "Redis read = token leak" to "Redis read + key access = token leak". Shorter retention is defense in depth. The flag-on-spec model means the decision lives next to the task definition where it's reviewable.

**Confidence**: high
**Reversibility**: moderate (deploy of the encryption layer; existing archived items remain plaintext until they age out)

---

### Q12-8: Plugin cron quota split
**Source**: doc 12 §20 (one rate-limit budget for cron + on-demand).

**Proposal**: **Split into two buckets per plugin**:
- `cron`: 10 tokens/min, refills at 10/min.
- `on_demand`: 60 tokens/min, refills at 60/min.

Both configurable per plugin by super_admin. The combined cap (70/min) is the same envelope as today's 60/min after accounting for a typical plugin's cron usage, but a plugin with a `@minute` cron no longer eats into its on-demand budget.

**Reasoning**: The doc itself names the problem (cron-heavy plugins struggle with on-demand). Splitting buckets is the obvious fix; the cron bucket's smaller size discourages plugins from using cron as a general-purpose scheduler. 10/min covers anything sub-minute (we don't allow that) up to a 6-second tick (we don't allow that either) — in practice it's "cron can fire up to 10 distinct scheduled tasks per minute," which is plenty.

**Confidence**: medium
**Reversibility**: cheap

## Doc 13 — Security Baseline

### Q13-1: Bug bounty pre-launch
**Source**: doc 13 §21 (pre-launch yes/no; cost vs signal-to-noise).

**Proposal**: **No bug bounty pre-launch.** Instead:
- P5: external pentest from a named firm (Trail of Bits / NCC Group / Doyensec class), 2-week engagement, scoped to the public API + admin + plugin sandbox.
- P5 + 3 months (post-GA): launch an **invite-only** bug bounty via HackerOne or Intigriti, capped at 20 researchers initially.
- P5 + 12 months: open to the public if triage capacity exists.

**Reasoning**: A pre-launch bounty produces a flood of low-signal reports (typo SSRFs, header-omission noise) at a time when the team has no triage capacity. Pentest finds the high-severity issues a focused engagement targets; bounty extends coverage to the long tail after the team can absorb the inbound rate. Invite-only is the well-established middle ground (Shopify, GitLab, Stripe all used it during their growth phase).

**Confidence**: high
**Reversibility**: cheap

---

### Q13-2: Site-wide Trusted Types rollout
**Source**: doc 13 §21 (extend from admin to public; theme cooperation).

**Proposal**: Three-phase rollout.
- **v1**: admin only (already specified).
- **v1.5**: opt-in via `theme.json` flag `security.trustedTypes: "enforce"`. Marketplace lists themes that opt in with a badge.
- **v2**: required for all marketplace themes; opt-out via deprecated grandfather list, sunset over one year.

**Reasoning**: Trusted Types requires every `innerHTML`/`document.write`/`eval`-style assignment to go through a policy. Legacy themes will not pass; forcing v1 site-wide is a launch blocker. The opt-in path lets motivated theme authors get an ecosystem badge in v1.5 (carrot), then the badge becomes the default in v2 (stick). One year sunset is the rhythm WordPress itself uses for security-breaking deprecations.

**Confidence**: high
**Reversibility**: moderate (the theme-manifest schema change is forward-compatible, but operators rely on the badge once it exists)

---

### Q13-3: SRI for public-site third-party scripts
**Source**: doc 13 §21 (once plugins declare SRI, require for any cross-origin script; oEmbed compatibility).

**Proposal**:
- **Plugin-declared cross-origin scripts**: SRI required from v1.5. Plugin manifest gains `assets.external[].integrity` field; missing it = plugin contract fail.
- **oEmbed provider scripts**: SRI **not** required (most don't publish stable hashes). Instead, oEmbed sources are restricted to a curated provider allowlist with TLS-pinned domains, and their scripts run in an `iframe-sandbox` with `allow-scripts` only (no `allow-same-origin`).
- **CSP report path**: warn on cross-origin script-src loads without SRI; admin dashboard surfaces the count.

**Reasoning**: SRI is binary: either every cross-origin script has it, or one missed source defeats the control. Plugin-declared scripts are author-controlled and SRI-able; oEmbed scripts are out of our control. Mixing the policies — strict where we can, sandbox where we can't — gets the protection without breaking embed UX. The CSP warning gives admins visibility into the gap.

**Confidence**: high
**Reversibility**: cheap

---

### Q13-4: Cosign keyless OIDC issuer
**Source**: doc 13 §21 (GitHub OIDC vs Anchore Fulcio vs self-hosted Sigstore).

**Proposal**: **GitHub OIDC for v1.** Build identity is the GitHub Actions workflow ID; signatures verify against `https://token.actions.githubusercontent.com`. No self-hosted Sigstore.

Trigger to revisit: if we ever move the build pipeline off GitHub (e.g., to GitLab or self-hosted Buildkite), re-evaluate at that point — likely Fulcio public for portability, self-hosted only if compliance demands.

**Reasoning**: Self-hosted Sigstore is real ops cost (Fulcio + Rekor + a CA pipeline). GitHub OIDC is the dominant choice in the Go ecosystem (cosign's own README leads with it), and our supply-chain trust root is already GitHub (the source lives there). Adding a separate identity provider doesn't reduce GitHub's compromise impact — if GitHub is compromised, our source is compromised before our signatures matter.

**Confidence**: high
**Reversibility**: cheap (signatures are per-release; switching issuer for future releases is straightforward)

---

### Q13-5: Reproducible builds commitment
**Source**: doc 13 §21 (v1 commitment level).

**Proposal**: **Best-effort for v1; hard goal for v2.** v1 ships:
- Pinned base images by digest.
- `SOURCE_DATE_EPOCH` set from the commit timestamp.
- Build provenance (§7.8) regardless of byte-reproducibility.

Hard requirement (byte-identical rebuild from public source + recorded inputs) shifts to v2 when the toolchain (Go, npm, container tooling) collectively makes it less painful.

**Reasoning**: True reproducible builds require fighting the entire toolchain (npm timestamps, Go embedded build info, container layer ordering). The marginal supply-chain benefit over signed-provenance-with-pinned-deps is real but incremental. Best-effort lets us claim the easy wins; hard goal for v2 sets the bar for when toolchain maturity catches up.

**Confidence**: high
**Reversibility**: cheap

---

### Q13-6: CSP report-only rollout
**Source**: doc 13 §21 (30-day report-only post-launch then enforce; long-tail theme risk).

**Proposal**: **Yes, 30-day report-only window per public-site release.** Specifically:
- **Public site CSP**: every release ships with `Content-Security-Policy-Report-Only` for the first 30 days post-deploy, then flips to enforcing `Content-Security-Policy`. The flip is automated via a release-day-+30 calendar event with a one-line config change.
- **Admin CSP**: **enforcing from day 1.** No report-only phase for admin — the user population is small (operators) and the trust requirement is higher.
- Report-only data feeds the CSP report endpoint (§3.5) and a release-blocking dashboard: more than 100 violations/hour on a single directive postpones the flip.

**Reasoning**: Themes are an open-ended source of CSP-breaking surprises; the report-only window catches them with real traffic. Admin is closed; we control every page that ships. The "delay flip if violations explode" guardrail prevents a hidden theme from going from broken-silently to broken-loudly on the same day.

**Confidence**: high
**Reversibility**: cheap

### Q13-7: Outbound HTTP allowlist defaults
**Source**: doc 13 §21 (ship a default allowlist of known-good providers).

**Proposal**: **Yes, ship a curated default allowlist that admins extend (never replace).** Defaults grouped by category:
- **Payments**: `api.stripe.com`, `*.paddle.com`, `api.lemonsqueezy.com`.
- **Comms**: `api.twilio.com`, `api.sendgrid.com`, `api.mailgun.net`, `api.postmarkapp.com`, `*.slack.com`.
- **Dev/SCM**: `api.github.com`, `gitlab.com`, `bitbucket.org`.
- **CDN/cloud**: `*.cloudfront.net`, `*.cloudflare.com`, `*.amazonaws.com` (S3 hosts).
- **oEmbed**: YouTube, Vimeo, Twitter/X, Instagram, TikTok official endpoints.

The list lives in `pkg/security/safehttp/defaults.yaml`, versioned, with each entry having a "last reviewed" date. Quarterly review by security-leads.

Admin UI surfaces "you are using <allowlist version X>; latest is Y" with a one-click upgrade.

**Reasoning**: The UX cost of typing 30 host allowlist entries is the kind of friction that drives admins to disable the allowlist entirely. A curated default is much safer than "all allowlists are operator-built". The quarterly review keeps the list honest; the versioning makes upgrades visible. "Never replace" matters: admins must extend the curated set, not opt out of it.

**Confidence**: high
**Reversibility**: cheap

### Q13-8: WAF placement
**Source**: doc 13 §21 (in-app L7 ruleset vs edge-only).

**Proposal**: **Ship an in-app L7 ruleset** in `pkg/security/waf`, OWASP CRS subset, opinionated and conservative (false-positive-averse). Edge WAF (Cloudflare, AWS WAF, etc.) is additive and operator-configured; not assumed.

Initial ruleset:
- SQL injection signatures on REST/GraphQL query params (CRS rule pack 942).
- Path traversal signatures on every URL-bearing param (CRS 930).
- Common log4j-style payload signatures, kept current via vuln-feed.
- Per-rule allowlist for the admin upload path (CRS 941 false-positives heavily on legitimate HTML content).

Findings emit `security.waf.matched_total{rule}` and a structured log entry; block decision is per-rule (some rules log-only).

**Reasoning**: Self-host is a first-class deployment target (doc 00); operators on a VPS without a CDN have no edge WAF. An in-app baseline gives every operator the same floor. The "log-only by default for new rules" pattern avoids the classic WAF rollout disaster (legitimate traffic blocked on a regex). Edge stays in the picture for those who run it, but we don't depend on it.

**Confidence**: high
**Reversibility**: moderate (once operators rely on the WAF metric, removing it requires deprecation)

### Q13-9: Anti-bot plugin slot
**Source**: doc 13 §21 (first-class slot for Turnstile / hCaptcha; affects 06 login).

**Proposal**: **Yes — first-class capability `bot_challenge`** in the plugin manifest. The capability grants a plugin the right to register a `challenge.verify(token) -> bool` host hook. The auth flow (06) and the comment-form path (§4.6 sanitization) call `challenge.verify` if any active plugin declares the capability.

Ship a reference Cloudflare Turnstile plugin in the marketplace at launch.

UI behavior:
- If zero `bot_challenge` plugins active: no challenge UI; rate limits and login bucketing carry the load (§11).
- If one or more active: challenge widget renders on login + comment + signup. Admins pick which one is authoritative (no double-challenge).

**Reasoning**: Bot defense is a moving target; building it into core would lock us to one vendor's challenge protocol. A capability slot lets the ecosystem track current best-of-breed (Turnstile today; whatever displaces it tomorrow). The reference plugin ensures parity with WordPress's "any anti-bot plugin works" UX.

**Confidence**: high
**Reversibility**: cheap

### Q13-10: Cookie-free anon sessions for public site
**Source**: doc 13 §21 (feasible; breaks CSRF and rate-limit accuracy).

**Proposal**: **Do not pursue.** Keep cookie-based anonymous sessions with strict privacy attributes:
- `SameSite=Lax`, `Secure`, `HttpOnly` where applicable, `Path=/`.
- No analytics or fingerprinting in the anon cookie — its sole content is a random ID for CSRF and rate-limit correlation.
- Disclosed in the privacy policy as a "strictly necessary" cookie (GDPR carve-out applies).

**Reasoning**: The question's own caveats are decisive — CSRF tokens need a server-side identity to bind to, and rate-limit accuracy is critically reduced by IP-only correlation (NAT, CGNAT, mobile carrier IPs). The privacy win of going cookie-free is small (a random anonymous ID is not PII) compared to the security cost. The "strictly necessary" cookie category exists for exactly this case.

**Confidence**: high
**Reversibility**: cheap

### Q13-11: `Cross-Origin-Opener-Policy: same-origin-allow-popups` for admin
**Source**: doc 13 §21 (popups exception worth the weaker isolation).

**Proposal**: **Default admin COOP stays `same-origin`.** Add `same-origin-allow-popups` **only on routes that initiate OAuth flows** — concretely `/admin/integrations/*/connect` and any plugin-declared OAuth route. Implementation via per-route middleware that overrides the default header.

**Reasoning**: OAuth popup flows are a real UX (connecting a Stripe/Mailchimp/etc. account), and the alternative (full-page redirect) loses the admin's draft state. Scoping the weaker policy to the small set of routes that need it preserves the strong default everywhere else. The blast radius of the exception is the popup window itself — a tradeoff worth taking on specific known-OAuth-initiating routes.

**Confidence**: high
**Reversibility**: cheap

### Q13-12: Pepper rotation automation
**Source**: doc 13 §21 (auto rotation vs operator-only).

**Proposal**: **Operator-only rotation, but the system nags.**
- A `security.pepper.age_days` metric is emitted.
- Alert at age >= 365 days: warning.
- Alert at age >= 540 days: critical (paged).
- `gonext secrets pepper rotate` is a single command that handles the dual-pepper window (old pepper kept for verifying existing hashes until all are migrated, per 06).

No auto-rotation, no rotation storms.

**Reasoning**: Pepper rotation requires application coordination (dual-pepper verification window) and storage migration cost. Auto-rotation either runs that migration unattended (risk: half-migrated state during an outage) or just bumps the version (risk: stale hashes that never re-verify). Operator-driven with strong nags gets the cadence right without the failure mode. The 365/540-day thresholds are arbitrary-but-reasonable; tune from incident data.

**Confidence**: high
**Reversibility**: cheap

### Q13-13: PII redaction in logs — write-time vs read-time
**Source**: doc 13 §21 (write-time safer; loses debug fidelity).

**Proposal**: **Write-time redaction.** The structured-log writer in `pkg/log` has a redaction policy applied before bytes hit the disk/network sink. Policy:
- A static field-name allowlist of known-sensitive keys (`password`, `token`, `secret`, `pepper`, `authorization`, `cookie`, `email`, `ip`, `user_agent`, `pii_*`).
- Email addresses: replaced with `<hash:first-8-of-sha256>` so the same email correlates across logs without being readable.
- IP addresses: replaced with `<network>/24` for IPv4, `/64` for IPv6.
- A `_redacted_fields` array attached to each log line so an operator knows what was scrubbed.

Lost fidelity is recovered via:
- A separately-scoped, encrypted, short-retention (3-day) "debug log" sink that captures unredacted records for sessions explicitly enabled by an operator with `manage_logs` capability. Used only for active incident investigation.

**Reasoning**: The doc's own framing names the right answer — write-time is the strong invariant ("PII never leaves the application boundary unredacted"). Losing fidelity is solvable by an explicit, audit-logged debug session — that's a workflow change, not a security regression. Read-time redaction is an unenforced wish that one stack trace, one panic-dump, one `log.Printf` away from breaking.

**Confidence**: high
**Reversibility**: expensive (once logs are written unredacted, the data lives forever in the SIEM)

### Q13-14: SOC 2 commitment for hosted offering
**Source**: doc 13 §21 (affects vendor selection).

**Proposal**: **Commit to SOC 2 Type II for the hosted offering, scoped to 12 months post-GA.** Implications for tooling choice today:
- **Logging**: pick a vendor with SOC 2 Type II report available (Datadog, Honeycomb, Sumo Logic) over self-hosted Loki — auditor friendly.
- **Secret store**: AWS Secrets Manager / GCP Secret Manager / HashiCorp Vault Enterprise — all have SOC 2 reports.
- **CI**: GitHub-hosted runners are SOC 2; self-hosted runners (we use them for heavy jobs per doc 11 §15.5) need our own controls documented.
- **Audit log**: append-only requirement (already specified in 06) maps directly to SOC 2 CC7 controls.

Self-host distribution is not in scope — operators run it on their own infra, their own compliance.

**Reasoning**: SOC 2 is the table-stakes enterprise contract gate. Committing now means tool choices made in P0-P3 are compliant-from-birth; bolting it on at P5 costs more (rip-and-replace of an analytics or logging vendor is multiple weeks of work). The 12-months-post-GA window is realistic for Type II, which requires 6+ months of observation. Type I earlier would be a stopgap; not committing to either leaves enterprise revenue on the table.

**Confidence**: medium (depends on go-to-market choices outside engineering's purview)
**Reversibility**: expensive (vendor lock-in)

### Q13-15: CSP nonce in proxy headers audit
**Source**: doc 13 §21 (`X-Script-Nonce` Go → Next.js: logged anywhere).

**Proposal**: **Ship a unit test today that asserts `X-Script-Nonce` is in the request-logger redaction set.** The test:
1. Builds a request with `X-Script-Nonce: test-nonce-deadbeef`.
2. Routes it through the request-logger middleware.
3. Asserts the captured log line does not contain `deadbeef`.
4. Asserts `_redacted_fields` contains `x-script-nonce`.

Wire the test into the lint stage (always runs). Any future code path that logs the header without redaction is caught by this test.

**Reasoning**: The question is fundamentally "did we forget anywhere"; a targeted test is cheaper than a one-shot audit because the audit needs to be re-run after every refactor. Adding the header name to the redaction set + a regression test is one PR's worth of work and a permanent invariant. Treat the discovery of any other leak path as a P0 — a leaked nonce defeats CSP for a session.

**Confidence**: high
**Reversibility**: cheap
